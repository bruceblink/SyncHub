# SyncHub v0.1.0 MVP Release Notes

## Summary

SyncHub v0.1.0 is the first MVP release for a personal developer workspace sync loop. It focuses on a local-first setup: SQLite metadata, local filesystem object storage, REST API, and CLI workflows including daemon-style sync cycles.

## Highlights

- API server with health, readiness, version, metrics, Swagger UI, and OpenAPI YAML endpoints.
- SQLite-backed auth, file metadata, upload sessions, change feed, devices, conflicts, and version history.
- Local filesystem storage with chunk upload, compose, full download, range download, ETag, and conditional download support.
- CLI support for auth, workspace init, manifest scan, file list/download/mkdir/move/delete, upload status, sync push/pull/once/status/doctor/watch/daemon/trash/devices/conflicts, and server diagnostics.
- CLI daemon support for registered workspace discovery, default local-change watching, scheduled sync fallback, one-shot sync, dry-run mode, pause/resume, status, reset-state, and JSON output.
- Multi-device sync smoke coverage for register, workspace init, push, pull, daemon control, version history, pin/unpin, restore, and download.
- Local backup and restore scripts for SQLite data plus object storage.
- Docker image release for Linux servers and Fly.io, deployment helper files, plus auxiliary Linux/Windows binary archives and SHA256 verification.

## Release Artifacts

The primary release artifact is the Docker image:

- `ghcr.io/bruceblink/synchub:0.1.0`
- `ghcr.io/bruceblink/synchub:v0.1.0`
- `ghcr.io/bruceblink/synchub:latest`

The image is published for:

- `linux/amd64`
- `linux/arm64`

The release also includes deployment helpers and auxiliary binary archives under `dist\synchub-0.1.0`:

- `docker-compose.release.yml`
- `fly.toml`
- `synchub-0.1.0-linux-amd64.tar.gz`
- `synchub-0.1.0-linux-arm64.tar.gz`
- `synchub-0.1.0-windows-amd64.zip`
- `SHA256SUMS.txt`

Each archive includes:

- `synchub-api`
- `synchub-cli`
- `README.md`
- `LICENSE`

`docker-compose.release.yml` is the Linux server deployment helper for the published image. `fly.toml` is the Fly.io deployment config for the same Docker image. Windows archives include `.exe` binary names for local development. Linux archives use `tar.gz` so Unix execute bits are preserved. These archives are fallback/manual-install artifacts; Docker is the intended deployment path.

## Docker Deployment

```bash
docker pull ghcr.io/bruceblink/synchub:0.1.0
docker run -d --name synchub-api \
  -p 8765:8765 \
  -e JWT_SECRET=change-me \
  -v synchub-data:/data \
  ghcr.io/bruceblink/synchub:0.1.0
```

Or use the release compose file:

```bash
export JWT_SECRET=change-me
export SYNCHUB_IMAGE=ghcr.io/bruceblink/synchub:0.1.0
docker compose -f docker-compose.release.yml up -d
```

Or deploy to Fly.io:

```powershell
# Edit fly.toml: set app name, primary_region, and image tag.
fly apps create synchub-your-name
fly volumes create synchub_data --app synchub-your-name --region nrt --size 1
fly secrets set --app synchub-your-name JWT_SECRET="replace-with-a-long-random-secret"
fly deploy --config .\fly.toml
```

## Known MVP Scope

The v0.1.0 release intentionally keeps the product narrow:

- SQLite is the supported MVP database path.
- Local filesystem storage is the supported MVP object storage path.
- Docker image deployment is the intended release path; Docker Compose and Fly.io use the same image.
- Fly.io MVP deployment is single-Machine only because Fly Volumes do not automatically replicate the SQLite database or local object storage.
- Docker Compose smoke testing may still be environment-dependent when registry access is unstable.
- WebDAV, S3/OSS/MinIO storage, team spaces, third-party adapters, and production PostgreSQL/MySQL adapters are deferred.
